## Page 02 | Executive Summary (결론 먼저)
현재 코드베이스는 컨설팅형 산출물 자동화 역량이 높고, `validate → render → qa → polish` 품질 파이프라인이 내장되어 있어 경영진 보고서 생산의 속도/일관성 측면에서 즉시 활용 가능하다. 다만 세아베스틸 요구 수준(약 30페이지)과 비교하면 추천 엔진이 20장 상한으로 제한되는 구조적 제약이 존재한다. 따라서 단기적으로는 20장 핵심 의사결정 덱 + 10장 상세 부록 체계로 운영하고, 중기적으로는 코드 상한 완화 및 제조(철강) 전용 분석모듈을 추가하는 투트랙이 최적안이다.

## Page 04 | 코드 현황 진단: 강점
강점은 네 가지다. 첫째, 파이프라인 오케스트레이션이 명확해 실무 반복 운영이 용이하다(`scripts/deck_cli.py`). 둘째, 품질 기준이 수치화되어 있어 산출물 검수가 자동화된다(`scripts/validate_spec.py`, `scripts/qa_ppt.py`). 셋째, 근거 연결(evidence) 자동 보강 기능으로 주장-출처 추적성이 확보된다(`scripts/enrich_evidence.py`). 넷째, 추천-분석-렌더가 CLI로 일원화되어 현업 확산 장벽이 낮다.

## Page 05 | 코드 현황 진단: 제약
핵심 제약은 다섯 가지다. 첫째, `scripts/recommend_strategy.py`에서 권장 슬라이드 수가 20장으로 강제되어 30페이지 경영진 보고서 구조와 불일치한다. 둘째, brief 파싱이 정규식 라벨 의존형이라 입력 문구가 조금만 달라도 점수 왜곡이 발생한다. 셋째, `scripts/analyze_client.py`의 sources 섹션 기대값이 고정돼 산업별 커스텀 섹션 반영이 제한된다. 넷째, 표/이미지 블록은 일부가 플레이스홀더 중심이라 시각 완성도 보정이 추가 필요하다. 다섯째, 외부 데이터 커넥터가 없어 수작업 데이터 적재가 전제된다.

## Page 06 | 벤치마크 결과: 기존 클라이언트 편차
동일 코드 기반에서도 운영 완성도 편차가 확인된다. `ecopro`는 Readiness 100, QA 경고 0으로 최고 수준이며, `acme-demo`도 99점으로 안정적이다. 반면 `posco-futurem`은 최근 분석 기준 57점, Spec 경고 29건으로 품질 편차가 크다. 시사점은 명확하다. 코드 자체보다 입력 아티팩트 품질(brief/sources/spec)과 운영 규율이 성과를 좌우한다. 세아베스틸 적용 시 기술 구현보다 운영 거버넌스 설계가 우선 과제다.

## Page 07 | 세아베스틸 현재 상태 (코드 실행 기준)
세아베스틸 신규 팩 기준 `recommend` 결과는 핵심 집중영역이 비용절감, 운영혁신, 데이터거버넌스, AI확산, 리스크통제 순으로 도출됐다. 그러나 `analyze` 결과 Readiness는 50점(Partial Ready)으로, `deck_spec.yaml` 미작성 상태가 직접 원인이다. 즉, 전략 방향성은 확보됐지만 실행 산출물(30슬라이드 스펙)로 전환되지 않은 상태다. 따라서 본 프로젝트의 1차 성공조건은 “전략 방향 합의”가 아니라 “스펙 완성 + QA 통과”다.

## Page 08 | 경영진 의사결정 프레임
의사결정은 3층으로 분리해야 한다. 1층은 재무성과(마진, 원가, 현금효율), 2층은 운영성과(수율, 납기, 품질, 재고), 3층은 전환역량(데이터 표준, AI 유스케이스 확산, 거버넌스)이다. 각 층은 독립 KPI로 측정하되 투자승인은 통합 게이트로 관리해야 한다. 통합 게이트 기준은 “재무효과 + 실행가능성 + 리스크 통제” 3요소이며, 어느 하나 미달 시 단계승인을 보류하는 원칙을 권고한다.

## Page 09 | 가치창출 레버 1: 수익성 복원
세아베스틸의 1순위 레버는 원가구조 개선과 믹스 최적화다. 실행 초점은 공정별 손실요인 분해, 재작업/불량 원인 상위 항목 집중, 에너지 원단위 최적화, 제품/고객 믹스 재정렬이다. 보고서 단계에서는 절대수치보다 “개선폭” 중심으로 설계하고, 본 실행단계에서 라인별 데이터로 정밀화한다. 권장 KPI는 영업이익률 개선(+1.5~2.5%p), 제조원가 3~5% 절감, 에너지 원단위 4~7% 절감이다.

## Page 10 | 가치창출 레버 2: 운영 안정화
운영 측면의 핵심은 본사와 현장 간 KPI 정의 통일이다. 동일 지표를 서로 다르게 산출하면 실행조직 간 갈등이 반복된다. 따라서 KPI 데이터 사전, 집계주기, 책임조직(RACI)을 먼저 고정해야 한다. 라인별로는 OEE 분해(가동/성능/품질), 납기 준수율, 재고회전일수, 클레임 재발률을 통합 모니터링 대상으로 묶고, 월간 운영회의에서 편차 원인과 보정 액션을 동시에 승인하는 운영리듬을 제안한다.

## Page 11 | 가치창출 레버 3: 데이터 기반 경영
현재 코드 파이프라인은 evidence 기반 보고 체계에 강점이 있으므로, 세아베스틸 데이터 거버넌스 설계와 궁합이 좋다. 권고사항은 세 가지다. 첫째, 데이터 도메인 오너를 지정해 지표 산식 분쟁을 차단한다. 둘째, 의사결정 핵심 지표만 우선 표준화해 과도한 범위 확장을 방지한다. 셋째, 경영회의 보고서와 운영 대시보드의 지표 정의를 동일화해 보고-실행 간 단절을 없앤다.

## Page 12 | 가치창출 레버 4: AI 유스케이스 확산
AI는 기술 실험이 아니라 재무효과 중심 포트폴리오로 운영해야 한다. 세아베스틸 우선 유스케이스는 품질 이상징후 조기탐지, 수요/생산/재고 동시 최적화, 설비고장 예측, 원재료 가격변동 시나리오 대응이다. 각 유스케이스는 “문제정의-데이터준비-모델실행-현장정착” 4단계 게이트를 통과해야 하며, 12개월 내 현장 정착률과 재무효과를 KPI로 관리해야 한다.

## Page 13 | 30페이지 보고서 아키텍처 제안
현재 코드 추천엔진은 20장을 제안하므로, 세아베스틸의 30페이지 요구는 아키텍처를 분리해 대응한다. 권장 구조는 20장 본편(결정용) + 10장 부록(근거용)이다. 본편에는 결정사항, 투자안, 로드맵, 리스크를 담고, 부록에는 가정/민감도/세부 KPI/데이터 정의를 배치한다. 이 구조는 임원 의사결정 집중도를 유지하면서 감사추적성과 실행력을 동시에 확보한다.

## Page 14 | 코드 개선 과제 1: 슬라이드 상한 조정
`scripts/recommend_strategy.py`의 `target_count = max(8, min(20, target_count))`는 본 과제의 직접 병목이다. 세아베스틸 기준으로 상한을 35로 조정하고, 20장 초과 시 자동으로 appendix 비중을 높이는 규칙을 추가해야 한다. 이렇게 하면 임원용 핵심 메시지는 간결하게 유지하면서도, 30페이지 요청을 코드 레벨에서 자연스럽게 수용할 수 있다.

## Page 15 | 코드 개선 과제 2: 산업별 템플릿 고도화
철강 제조 특화 모듈이 필요하다. `analyze_client.py`의 제조 addon(X1)을 확장해 탄소/에너지/원재료 변동/고객품질 요구를 포함한 steel 전용 모듈(X2)을 추가하는 것이 바람직하다. 동시에 `layout_preferences`에 철강 의사결정 관점의 비교/시나리오/공정흐름 레이아웃 프리셋을 제공하면 보고서 제작 리드타임을 크게 단축할 수 있다.

## Page 16 | 코드 개선 과제 3: 입력 안정성 강화
brief 파싱을 정규식 라벨 기반에서 YAML 구조 기반으로 전환해야 입력 변동에 강해진다. 현재는 라벨 문구 차이로 Audience가 누락되는 현상이 실제 발생한다. 또한 sources 섹션 기대값(`market/client/competitors/tech-trends`)을 설정파일로 분리해 산업별 커스터마이징을 허용해야 한다. 이를 통해 점수 왜곡을 줄이고, 경영진 신뢰도를 높일 수 있다.

## Page 17 | 코드 개선 과제 4: Value Case 자동화
세아베스틸 과제의 핵심은 재무 정당성이다. 따라서 Value Case 산정 스크립트를 별도 모듈로 분리해 시나리오(보수/기준/공격)별 효과를 자동 계산하도록 권고한다. 입력은 KPI baseline, 개선율 가정, 투자비, 램프업 기간이며, 출력은 EBITDA 개선, 현금효과, payback 범위다. 현재 프레임워크에 이 모듈이 추가되면 C레벨 투자의사결정 속도가 크게 개선된다.

## Page 18 | 실행전략 Wave 0 (4주): Baseline 고정
첫 4주 목표는 분석 정확도보다 기준선 합의다. 산출물은 공통 KPI 사전, 데이터 소스 매핑, 의사결정 RACI, 30페이지 스토리라인 초안이다. 기술 관점에서는 `deck_spec` 최소 골격(30슬라이드)을 먼저 채우고, evidence 앵커를 우선 연결한다. 운영 관점에서는 CFO/COO/CIO 공동 승인회의를 주 1회 고정해 범위변경을 통제한다.

## Page 19 | 실행전략 Wave 1 (8주): 핵심 Use Case 파일럿
파일럿은 2~3개로 제한한다. 권장 조합은 품질예측, 에너지최적화, 생산-재고 동기화다. 각 파일럿은 효과지표를 사전에 합의하고, 효과 미달 시 중단 기준을 명시해야 한다. 본 단계의 성공조건은 기술 데모가 아니라 KPI 개선 실증이다. 문서화는 `strategy_report`, `analysis_report`, `deck_spec`을 동기화해 경영진과 현장이 동일 메시지를 공유하도록 설계한다.

## Page 20 | 실행전략 Wave 2 (6개월): 확산 및 제도화
파일럿 성공 후 전사 확산 단계에서는 표준 운영체계를 고정해야 한다. 표준 항목은 데이터 품질점검 주기, 모델 재학습 정책, 현장 반영 승인 절차, 리스크 대응 매뉴얼이다. 동시에 월간 ExCom 리뷰에서 “효과-리스크-투자” 3축 대시보드를 고정 의제로 운영한다. 코드 측면에서는 full-pipeline 자동실행과 QA 보고서 누적 관리를 통해 품질 편차를 통제한다.

## Page 21 | 재무 Value Case (범위형)
내부 데이터가 partial 상태이므로 본 보고서는 범위형 가치산정을 적용한다. 기준 시나리오에서 원가 3~5% 절감, 납기준수율 +4~6%p, 불량/클레임 20~30% 감축을 목표로 한다. 보수 시나리오는 효과의 60~70%, 공격 시나리오는 120~140%를 가정한다. 투자승인은 절대규모보다 payback 기간과 실행리스크를 함께 판단해야 하며, 12~18개월 내 회수 가능성을 1차 기준으로 권고한다.

## Page 22 | 리스크 관리 프레임
리스크는 기술보다 조직/운영에서 크게 발생한다. 핵심 리스크는 KPI 정의 충돌, 데이터 품질 저하, 현장 채택 저항, 투자우선순위 갈등, 규제 대응 지연이다. 대응 원칙은 단순하다. 리스크별 오너를 지정하고, 월간 리스크 리뷰에서 잔여리스크와 완화조치를 정량 점검한다. 보고서에는 리스크 열거가 아니라 “발생조건-감지지표-즉시조치”까지 포함해야 실행력이 확보된다.

## Page 23 | 거버넌스 설계
권장 거버넌스는 3계층이다. ExCom(월간)에서 투자/우선순위 승인, Transformation PMO(주간)에서 과제 진행관리, 현장 워킹그룹(상시)에서 데이터/프로세스 개선을 실행한다. CFO는 효과 검증, COO는 운영 성과, CIO는 데이터/시스템 안정성 책임을 가진다. 의사결정 지연을 막기 위해 승인권한 매트릭스를 사전 합의하고, 분쟁 사안은 2주 내 상위 의사결정으로 자동 승격하는 규칙을 둔다.

## Page 24 | KPI 체계 및 품질게이트
KPI는 재무/운영/전환 3층으로 통합 설계한다. 재무: 영업이익률, 원가율, 현금전환. 운영: 수율, 납기준수율, 불량률, 에너지 원단위. 전환: 데이터 표준화율, AI 적용 라인 수, 현장 채택률. 품질게이트는 코드 기준과 정합시킨다. evidence 커버리지 80% 이상, QA 오류 0, 레이아웃 밀도 규칙 준수, 민감도 가정 명시를 필수 통과 조건으로 적용한다.

## Page 25 | 보고체계 운영안 (C-Level)
임원 보고는 월 1회 정례, 분기 1회 심화리뷰를 권고한다. 월간 보고서에는 KPI 편차와 즉시조치 중심의 10장 내외 요약을 제공하고, 분기 보고서에는 30페이지 풀 패키지를 제출한다. 이때 본편과 부록을 분리해 회의 중 의사결정 피로를 줄여야 한다. 본 코드 기반 자동화는 분기 보고서 제작 리드타임을 단축하는 데 강점이 있으므로, 정례 운영체계에 직접 편입하는 것이 효과적이다.

## Page 26 | 12주 실행 백로그 (우선순위)
우선순위 1은 `deck_spec` 30슬라이드 완성과 evidence 연결률 확보다. 우선순위 2는 전략추천 상한 확장(20→35)과 steel 전용 모듈 추가다. 우선순위 3은 Value Case 자동산정 스크립트 도입이다. 우선순위 4는 QA 리포트 누적 대시보드 구축이다. 각 과제는 기능완료보다 운영 정착을 완료 기준으로 정의해야 하며, 파일 생성 자체를 성과로 간주하면 안 된다.

## Page 27 | 조직 변화관리 계획
변화관리의 핵심은 현장 저항 최소화다. 초기에는 KPI가 성과평가 수단으로 오해될 수 있으므로, 목적을 “개선지원”으로 명확히 커뮤니케이션해야 한다. 또한 본사 주도의 일방 설계가 아니라 공장 리더를 공동 오너로 지정해 설계 수용성을 높인다. 교육은 도구 사용법보다 “의사결정 프레임” 중심으로 진행하고, 초기 성공사례를 빠르게 가시화해 확산 동력을 만든다.

## Page 28 | 예상 성과 및 경영진 메시지
본 제안이 계획대로 실행되면 단기적으로는 원가/품질/납기 지표 개선, 중기적으로는 데이터 기반 의사결정 전환이 가능하다. 경영진이 기억해야 할 메시지는 세 가지다. 첫째, 기술보다 운영규율이 성패를 좌우한다. 둘째, 30페이지 보고서는 목적이 아니라 의사결정 품질을 높이는 수단이다. 셋째, 투자승인은 단계게이트 방식으로 관리해야 리스크를 통제할 수 있다.

## Page 29 | C-Level 의사결정 요청사항
즉시 의사결정이 필요한 항목은 다섯 가지다. 1) 12주 진단-설계 착수 승인. 2) CFO/COO/CIO 공동 스폰서 체계 확정. 3) 파일럿 3개 유스케이스 범위 승인. 4) KPI 표준 정의 원칙 승인. 5) 코드 개선(슬라이드 상한/산업모듈/Value Case 자동화) 투자 승인. 이 다섯 항목이 합의되면 실행 전개 속도가 급격히 올라가며, 반대로 지연 시 보고서 품질은 유지돼도 성과 창출은 지연된다.

## Page 30 | 결론 및 다음 단계
세아베스틸 관점에서 현재 코드베이스는 “즉시 활용 가능한 기반”이며, 보고 품질 자동화 역량은 이미 충분하다. 다만 30페이지 C레벨 요구와 철강산업 특수성을 완전 충족하려면 추천엔진 상한, 산업모듈, Value Case 자동화의 3개 축 보강이 필요하다. 권고 실행순서는 `입력정합성 확보 → 30슬라이드 spec 완성 → full-pipeline QA 통과 → 파일럿 가치검증 → 전사확산`이다. 이 순서를 지키면 단기 성과와 중장기 체질 전환을 동시에 달성할 가능성이 가장 높다.
